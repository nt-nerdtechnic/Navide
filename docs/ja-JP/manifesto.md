# Navide マニフェスト

[English](../en-US/manifesto.md) | [繁體中文](../zh-TW/manifesto.md) | 日本語 | [ドキュメント](README.md)

## エンジニアリングを動かす新しい力

蒸気機関は、単に一人ひとりの労働を速くしただけではありません。生産力を人間の筋力から切り離し、新しい機械、工場、交通システム、組織、生活様式を可能にしました。

人工知能は、知的労働に同じような断絶を生み出しつつあります。ソフトウェアエンジニアリングにおける実行力は、もはや一人が入力し、記憶し、順番に完了できる量だけに制限されません。一人のエンジニアが複数の人工 Agent を指揮し、プロダクトの調査、計画、実装、テスト、レビュー、改善を並行して進められます。

AI モデルは新しいエンジンです。しかし、エンジンだけでは産業システムは生まれません。信頼できる機械、制御、記憶、フィードバック、運用方法を通じて人間がその力を指揮できて初めて、変革が起こります。

Navide は、そのためのエンジニアリングツールとして作られています。

> **一人のエンジニア。AI エンジニアリングチーム全体の力。**

## 従来の IDE を越えて

従来の IDE は、エンジニア自身が実装の大部分を行うという根本的な前提の上に設計されました。その中心には、ファイル、Editor、手入力する Command があります。

Agent 時代は、エンジニアリング作業の中心を変えます。

エンジニアは、意図を定義し、方向を選び、並行する知性を組織し、例外を解決し、Trade-off を判断し、根拠を受け入れる役割をますます担います。コード編集は引き続き不可欠ですが、ソフトウェアシステムとの唯一の、あるいは常に主要な対話ではなくなります。

```text
従来の IDE
エンジニア → ファイルを編集しツールを操作する → ソフトウェア

Navide
エンジニア → 目標、Agent、意思決定、根拠を指揮する → ソフトウェア
```

Navide は、古いモデルにチャットボックスを追加することを目指していません。その後に来る作業環境を確立しようとしています。

## 協調が新しいボトルネックになる

実行力が不足していた時代、エンジニアの主な制約は、コードを順番に理解し、書き、テストできる量でした。多くの Agent が同時に実行できるようになると、別の制約が支配的になります。

- それぞれの成果を誰が担当するのか
- 各 Session にどのような文脈と境界が必要か
- どの Task が独立して進められ、どれが待つべきか
- どこで意思決定、変更、仮定が衝突しているか
- いつ人間の注意を戻すべきか
- どの根拠が成果を信頼できるものにするか

協調のない知性の増加は、動きを増やしても、必ずしもエンジニアリングを改善しません。したがって Agent 時代には、より優れたモデルだけでなく、因果関係、品質、人間の責任を手放さずに並行作業を指揮する運用環境が必要です。

## 一人の人間をエンジニアリング組織へ

産業組織は、人員、部門、管理層、会議、Process を増やして規模を拡大してきました。AI は別の形を可能にします。一人の意図を中心に動的なエンジニアリング組織を組み立てる形です。

Navide のユーザーは、提案を受け取るだけのプログラマーではありません。AI エンジニアリングチームを率いる存在です。Agent が Product Analysis、Architecture、Implementation、Testing、Review、Maintenance を担う一方で、エンジニアは目的、制約、判断、受け入れに責任を持ち続けます。

目標は人間を排除することではありません。一人のアイデアと、それを実現するために歴史的に必要だった組織能力との間の制約を取り除くことです。

## Management by Exception

エンジニアが、可逆的なファイル編集や定型的な Test Command のたびに承認する必要はありません。それでは強力なシステムが、遅い Prompt の連続に戻ってしまいます。一方で、Agent が不可視かつ制御不能な Black Box として動くべきでもありません。

Navide は別の原則に従います。

> **高い自律性、継続的な可視性、即時の中断可能性、そして根拠に基づく説明責任。**

エンジニアは目標、境界、優先順位、不可逆な決定を設定します。Agent はその境界内で進みます。曖昧さ、リスク、競合、外部への影響、人間の判断によって介入の価値が生じたとき、Navide はエンジニアへ注意を戻します。

## 非公開の Project Intelligence

プロジェクトは Source File だけではありません。意図、意思決定、未完了の作業、Session History、失敗した試行、Test Evidence、Handoff、蓄積された理解があります。

Navide は、この個人的なエンジニアリングメモリを `.agent-team/` にローカル保存します。これは個々のエンジニアに属し、Hosted Navide Account にも、共有 Git Repository にも属しません。新しい Session や異なる Agent が文脈を継承できる一方で、非公開 Prompt、Token Data、Terminal History をチームに公開される Source Artifact に変えません。

Source Code はソフトウェアプロジェクトの共有された事実であり続けます。`.agent-team/` は、それを指揮するエンジニア個人の Intelligence です。

## 宣言より先に証明を

革命的な方向性は、曖昧な主張を正当化しません。Navide は、現在動くものと、目指しているものを区別しなければなりません。

最初の証明は、Navide が Navide の開発に使われていることです。次の証明は、他のエンジニアが自身の実プロジェクトで実作業を指揮することから得る必要があります。置き換え、自律性、速度、安全性、導入に関する主張は、プロダクトの挙動と独立した根拠が裏付ける範囲でのみ拡大すべきです。

基準は単純です。Workflow を示し、境界を公開し、成果に根拠を添えることです。

## 私たちの方向性

Navide は、完全な生産ループをデジタルに表現し検証できるソフトウェアエンジニアリングから始めます。要件は Task になり、Agent が Code を変更し、Test が根拠を生み、Git が成果を記録し、動くソフトウェアを実行できます。

Mission は明確です。

> **Agent 時代のエンジニアリングツールを構築する。**

エンジニアがアイデアから始め、AI エンジニアリングチームを創造と継続的な進化へ導き、必要なときだけ正確に介入し、従来の IDE を主要環境へ戻すことなく信頼できるソフトウェアを届けられたとき、Navide は成功します。
