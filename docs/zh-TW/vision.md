# 產品願景

[English](../en-US/vision.md) | 繁體中文 | [文件中心](README.md)

## 願景

Navide 將成為 Agent 時代的主要工程環境：一套讓一位工程師能在完整軟體生命週期中指揮、同步並驗證 AI 工程力量的系統。

## 任務

在保留人類意圖、判斷、所有權與最終責任的前提下，讓一位工程師擁有整個軟體團隊的執行能力。

## 產品類別

Navide 是一套 AI-native 軟體工程環境：Agent 時代的工程利器。

這個類別比 AI Editor、Terminal Multiplexer 或一組聊天 Session 更完整。Navide 圍繞成果、Agent、Session、協調、私有專案智慧、人類例外與驗收證據來組織工程工作。檔案、Editor、Terminal 與 Git 仍然是環境中的第一級工具。

## 目標使用者

Navide 首先為同時管理產品與多個 AI Agent 的個人開發者而打造，包括獨立開發者、技術創辦人，以及負責完整產品範圍的工程師。

產品不打算成為所有知識工作的通用作業系統。它的領域是軟體工程。

## 產品論點

傳統 IDE 圍繞檔案與直接輸入程式碼組織工作；AI-native 工程環境則必須圍繞以下項目組織：

- 目標與驗收條件
- Agent、角色、能力與 Session
- 任務、依賴關係、所有權與進度
- 共用脈絡、決策與 Handoff
- 變更、測試、審查與證據
- 人類例外與最終驗收
- 能跨越個別 Session 的私有專案智慧

Editor 仍然是第一級工程工具，但它成為更大系統中的一種 Intervention 模式，而不是開發工作的唯一中心。

AI 提高執行能力，協調因而成為新的瓶頸。Navide 的存在，是為了讓並行 Agent 工作具備明確所有權、足夠脈絡、全程可見、可中斷、可恢復並可驗證。

## 三種工程循環

### Genesis

將想法轉化為產品的第一個可運作成果。可設定的 Pipeline 從需求推進到規劃、設計、實作、審查與測試。成果是一個經過驗證的 Prototype 與一套可延續的專案脈絡，而不是開發工作的終點。

### Evolution

持續發展既有專案。工程師設定下一項功能、修正、實驗或品質目標；Navide 協調一個或多個 Session 進行實作、測試、修正與驗證。這是產品日常使用的中心。

### Intervention

當需要判斷或精準操作時，讓工程師直接檢查並修改成果。Diff、Editor、Terminal、Diagnostics、Git、Tests 與 Review 並不是失敗時的備案，而是刻意保留的人類控制工具。

## 運作哲學

Navide 採用例外管理。

### 工程師負責

- 產品意圖與優先順序
- 重要限制與品質標準
- 具有長期影響的架構或產品選擇
- 憑證、付款、發布、部署與破壞性操作
- 交付成果的最終驗收

### Agent 在核准邊界內負責

- 研究與本機 Codebase 探索
- 規劃與任務拆解
- 可逆的 Workspace 修改
- 例行 Command、測試、診斷與修正
- 進度更新、結構化 Handoff 與證據收集
- 反覆執行直到驗收條件通過或遇到例外

### Navide 負責

- Agent 與 Session 生命週期
- 任務路由與平行協調
- 脈絡同步與私有專案記憶
- 衝突、風險、閒置與失敗偵測
- 可見性、中斷、恢復與可稽核性
- 在正確時間將正確例外交還給工程師

## Project Intelligence Layer

`.agent-team/` 是 Workspace 的本機、個人智慧層。未來模型可能包含專案狀態、Session、Run、任務、決策、Handoff、證據與協調 Metadata。

這一層：

- 屬於個別使用者
- 預設儲存在本機
- 排除於 Git
- 與原始碼及團隊共用專案文件分離
- 只能透過明確、受控的 Export／Import 流程攜帶
- 受到保留、遮蔽與刪除控制

它絕不能成為隱含的雲端帳號、隱藏的團隊同步機制，也不能成為將憑證放入 Agent Context 的藉口。

## North-star 體驗

工程師應該能夠：

1. 開啟專案並立即恢復私有工程脈絡。
2. 描述下一個成果，而不是手動準備每一個實作步驟。
3. 看到 Navide 建議或啟動正確的 Agent 與 Session。
4. 讓獨立工作平行進行，而不發生無聲重疊或脈絡遺失。
5. 只在有意義的例外出現時接收注意通知。
6. 檢查推理軌跡、變更、測試、風險與未解決決策。
7. 隨時透過 Editor、Terminal、Diff 或直接指示進行介入。
8. 接受具有證據支持的成果，並繼續下一個 Evolution 循環。

## 「取代」的意義

取代傳統 IDE 並不表示逐項複製其介面功能，而是讓工程師能夠在不以另一套 IDE 為主要環境的情況下，完成理解、建立、瀏覽、編輯、執行、除錯、測試、審查、版本控制與交付的完整專業工作流。

Navide 最終必須提供完整工程能力，但每項能力都應該以 Agent 協調與人類判斷重新詮釋，而不是未經思考地繼承程式碼輸入時代的模式。

## 成功定義

當以下情況成立時，Navide 才算成功：

- 使用者選擇它作為日常主要工程環境。
- 個人工程師能持續管理多個有效率的 Agent Session，而不失去控制或脈絡。
- 產品 Evolution 變得更快，同時不讓品質、安全或因果關係變得不透明。
- 私有 Project Intelligence Layer 讓每個新 Session 都比孤立對話更有效。
- 工程師把更多注意力放在方向、架構、判斷與驗收，而不是重複執行。
- 完整產品能從想法一路走向持續交付，不必回到傳統 IDE 工作流。

## 證據紀律

Navide 的企圖必須大於目前實作，但不能讓兩者混為一談。

- README 的能力清單與 User Guide 描述目前可用行為。
- Roadmap 描述方向性成果與 Exit Criteria，而不是交付承諾。
- 創辦人使用 Navide 開發 Navide 是有效的 Dogfooding 證據，不是客戶驗證。
- 關於取代、自治、生產力、安全、採用與效能的宣稱，都必須具有與其強度相符的產品或獨立證據。
- 已知的安全、隱私、發布與平台邊界必須持續出現在公開溝通中。

正式的溝通層次與主張邊界請見[產品定位與公開主張](product-positioning.md)。
