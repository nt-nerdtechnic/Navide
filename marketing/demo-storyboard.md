# Navide Real-Product Demo Storyboard

## Purpose

Replace the conceptual workflow model on the landing prototype with a truthful recording that shows Navide's daily Evolution loop. The viewer must see the input, parallel execution, a meaningful human exception, and verified output.

## Recording task

Use a small, real improvement to Navide itself. The preferred task is a contained documentation or UI-quality change that:

- Has one clear acceptance criterion
- Can be divided across two or three sessions
- Produces a visible result
- Includes at least one test, review, or other evidence artifact
- Does not require credentials, deployment, destructive operations, or private customer data

Do not stage a fake customer project. A prepared Navide branch with scrubbed local state is acceptable.

## Required preparation

1. Create a clean demonstration workspace and branch.
2. Remove unrelated `.agent-team/` history or use a fresh local workspace copy.
3. Disable notifications that may reveal private information.
4. Use non-sensitive filesystem names and terminal prompts.
5. Close unrelated sessions, browser tabs, issue content, and Git remotes.
6. Confirm the task can be completed once without recording.
7. Capture at a stable resolution with readable interface scale.

## 45-60 second hero edit

| Time | Product action | On-screen message |
|---|---|---|
| 00:00-00:05 | Open the Navide workspace and recover its sessions | `One project. Its engineering context intact.` |
| 00:05-00:12 | State the next outcome and acceptance criteria | `Define the result, not every keystroke.` |
| 00:12-00:22 | Start or reveal two to three independent agent sessions | `Parallel execution. One visible system.` |
| 00:22-00:32 | Show progress, handoff, or Pipeline movement | `Context moves with the work.` |
| 00:32-00:42 | Surface one real exception or conflicting choice | `Your attention returns where judgment matters.` |
| 00:42-00:52 | Resolve through instruction, Diff, terminal, or mini IDE | `Intervene with precision.` |
| 00:52-00:60 | Show passing evidence and final result | `Accept outcomes backed by evidence.` |

## Two-to-three-minute narrated version

### Opening

> This is Navide running the Navide project. It is the primary environment used to evolve the product—not a chat panel beside another IDE.

### Goal

> I begin with an engineering outcome and acceptance criteria. Navide keeps the work organized around that result.

### Dispatch

> Independent sessions can research, build, and review in parallel. Their lifecycle, activity, and project context remain visible in one workspace.

### Exception

> The system is not designed to hide engineering judgment. When an assumption, risk, or conflict matters, attention returns to the engineer.

### Intervention

> I can redirect the agent or intervene through Diff, editor, terminal, diagnostics, tests, and Git. Direct editing remains first-class, but it is no longer the only way work moves forward.

### Evidence

> The result is accepted only after the change and its evidence are visible. That is the operating model: highly autonomous, continuously visible, immediately interruptible, and accountable through evidence.

## Capture list

- Workspace opening and recovered sessions
- Goal or Pipeline configuration
- Multiple active panes with clearly different responsibilities
- History, handoff, or run context
- A real exception or review point
- Diff or editor intervention
- Test or review evidence
- Final Git state or working UI result

Export:

- `hero-workflow.mp4` — 45-60 seconds, no narration required
- `product-tour.mp4` — 2-3 minutes, narrated
- `workflow-loop.webm` — silent 8-12 second loop for the hero
- Three still frames at desktop resolution

## Privacy review

Review every frame before publication for:

- Credentials and environment variables
- Usernames and home-directory paths
- Private prompts, session IDs, or `.agent-team/` history
- Token or billing information
- Unrelated repositories, issues, branches, or remotes
- Notification content
- Third-party data or copyrighted material

## Acceptance checklist

- A viewer can identify the task without narration.
- At least two distinct execution paths are visible.
- The human decision is consequential, not theatrical.
- Evidence appears before acceptance.
- No screen implies capabilities that were simulated outside Navide.
- The final edit does not contain private data.
